Analyze the simplified TypeScript AST module definitions in `/app/data/modules/` (one JSON file per module) and produce a complexity audit report at `/app/output/complexity_audit.json`. The project manifest at `/app/data/project.json` lists the module files to process. Metric thresholds are in `/app/config/thresholds.json` and severity/risk configuration is in `/app/config/weights.json`. A TypeScript on Node 22 starter is at `/app/src/main.js`; implement your solution in TypeScript. Do not modify any file under `/app/data/` or `/app/config/`.

Each module JSON file contains `module_name`, `path`, `lines_of_code`, `imports`, `exports`, and `functions`. Each function has `name`, `lines`, `parameters`, `body` (a tree of control-flow AST TypeScript on Node 22s with types like `if`, `else_if`, `else`, `for`, `while`, `switch`, `case`, `catch`, `ternary`, `try`, and logical operators in `condition` arrays), `operators`, and `operands`.

For each function compute cyclomatic complexity, cognitive complexity, Halstead metrics, and maintainability index. For each module compute coupling metrics from internal import references. Generate findings when metrics exceed configured thresholds and compute risk scores. The precise definitions, formulas, and edge-case handling for cyclomatic vs cognitive complexity counting rules are in `/app/docs/COMPLEXITY_SPEC.md`; Halstead metric formulas in `/app/docs/HALSTEAD_SPEC.md`; maintainability index formula in `/app/docs/MAINTAINABILITY_SPEC.md`; coupling computation rules in `/app/docs/COUPLING_SPEC.md`; finding generation, risk score formula, and sorting rules in `/app/docs/FINDINGS_SPEC.md`. Read those specification documents carefully — they are the authoritative source for all computation rules. Write the report with the exact structure, key sets, sort orders, and formatting defined in `/app/docs/OUTPUT_FORMAT.md`. Create the `/app/output/` directory if it does not exist.
