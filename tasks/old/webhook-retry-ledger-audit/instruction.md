Platform teams need a frozen snapshot of outbound webhook subscription health after rolling delivery windows, tier-specific chargeable outcomes, scheduled-day grace slips, signing-key rotation lag, incident-driven retry-budget deltas (which may net negative for a tier), cross-event failure-day suppression with per-pair deduplication, previous-window carryover gated on the sign of the tier's net delta, bronze surge pressure, gold endpoint throttle penalties, and explicit operator force or endpoint compromise directives are all applied on the same timeline anchored to `pool_state.current_day`. The outcome is a reproducible audit trail a downstream system can ingest without reprocessing raw delivery logs.

The system delivers five UTF-8 JSON documents under `/app/audit/` named `subscription_verdicts.json`, `tier_retry_budgets.json`, `incident_journal.json`, `endpoint_touchpoints.json`, and `summary.json`. Field names, ordering rules, incident kinds, disposition values, and every numeric relationship tied to the pool day are defined in `/app/webhooks/SPEC.md`, including the canonical JSON presentation each artifact must follow.

Technical formatting and paths: bundled evidence lives under `/app/webhooks/`. A non-empty `WRLA_DATA_DIR` overrides the input root; a non-empty `WRLA_AUDIT_DIR` overrides the output directory. When a variable is unset or empty, that side falls back to `/app/webhooks/` for reads and `/app/audit/` for writes. The output directory is created when missing. The default `/app/webhooks/` tree is not modified by the audit run.
