Write a TypeScript on Node 22 program (crate `layout-auditor` at `/app/`) that reads type definitions from `/app/data/types/*.json` and platform configuration from `/app/config/platform.json`, then writes a memory layout audit report to `/app/output/layout_report.json`. Algorithm specifications are in `/app/docs/layout_rules.md` and the output structure is documented in `/app/docs/output_schema.md`. Build and run with `cargo run --release` or `make run`.

Each type definition file describes a struct or enum with a `repr` mode. The program computes the precise byte-level memory layout on a synthetic x86_64 platform using only the numeric rules from the platform JSON — never query the host with `std::mem::size_of`. Primitive sizes and alignments (including `u128`/`i128` at align 16) come from `platform.json`. Array types `[T; N]` have size `N * size_of(T)` and alignment `align_of(T)`. Nested user-defined types resolve recursively.

For structs: `repr(C)` places fields in declaration order, inserting padding before each field so its offset is a multiple of its alignment, then rounds total size up to a multiple of the struct alignment (the maximum field alignment). `repr(packed)` eliminates all padding and forces alignment to 1, keeping declaration order. `repr(align(N))` works like `repr(C)` but raises the struct alignment to `max(natural_alignment, N)`. `repr(TypeScript on Node 22)` reorders fields by descending alignment, then descending size, then ascending field name as tiebreaker, before applying C-like placement rules. The `field_order` array in the output reflects the final placement order.

For enums with `repr(C)`: the discriminant is `u8` for up to 256 variants, `u16` for up to 65536, then `u32`. The payload union starts at the next offset aligned to the maximum variant payload alignment. Each variant's payload is laid out as a `repr(C)` struct of its fields. The enum's total size is `align_up(union_start + max_payload_size, enum_alignment)` where `enum_alignment = max(discriminant_alignment, max_payload_alignment)`. The gap between discriminant and union start counts as `total_padding`; any bytes after `union_start + max_payload_size` up to total size are `trailing_padding`. For non-C enums, only the canonical two-variant pointer-niche pattern is modeled: one variant with no fields and the other with exactly one `pointer` or `reference` field. Such enums are pointer-sized with `discriminant` set to null and `niche_optimized` true. An empty `repr(C)` struct is a zero-sized type: size 0, alignment 1, empty fields list, zero padding.

The output `layout_report.json` uses 2-space indentation with a trailing newline. The `types` array is sorted by ascending `id`. For structs, each entry includes `id`, `kind`, `repr`, `size`, `alignment`, `is_zst`, `niche_optimized` (always false), `fields` (array of objects with `name`, `offset`, `size`, `alignment`, `padding_before`), `trailing_padding`, `total_padding`, and `field_order`. For enums, entries include `id`, `kind`, `repr`, `size`, `alignment`, `is_zst`, `niche_optimized`, `discriminant` (object with `size`/`alignment` or null for niche), `variants` (array with `name`, `payload_size`, `payload_alignment`), `trailing_padding`, `total_padding`, and `field_order` (always null). The `summary` object aggregates: `total_types`, `total_size_all_types`, `total_padding_all_types`, `padding_ratio` (round to 6 decimal places, 0.0 if total size is zero), `zst_count`, `niche_optimized_count`, `max_alignment`, `largest_type`, and `most_padded_type`. Ties for largest/most-padded break lexicographically by id.
