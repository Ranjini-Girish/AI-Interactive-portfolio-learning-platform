The yard audit CLI in `/app` is giving planners bad status counts. Please fix the Go code so this works reliably:

`go run ./cmd/yardcheck --manifest /app/data/manifest.csv --appointments /app/data/appointments.csv --events /app/data/events.ndjson --out /app/out/yard_report.json`

The report must be deterministic JSON with `generated_at`, a `summary` count for `ready`, `blocked`, `stale`, `overbooked`, and `unknown_appointment`, a `bookings` audit summary, a `carrier_summary` audit summary, an `exceptions` ledger, and one `containers` entry per manifest container sorted by `container_id`. Set `generated_at` to the latest event timestamp in RFC 3339 UTC. The manifest has `container_id`, `booking`, `carrier`, and `requires_temp_check` columns. Each container entry must contain `container_id`, `booking`, `carrier`, `status`, `reason_codes`, `appointment_window_utc`, `last_event_type`, and `last_event_utc`; `appointment_window_utc` is `[start_utc, end_utc]` in RFC 3339 UTC, or `[]` when the booking has no appointment.

Appointment windows are local wall-clock times in the timezone column and are evaluated in UTC with inclusive starts and exclusive ends. Appointments also have `capacity` and `carrier_cutoff_min`; evaluate the carrier cutoff only after confirming the latest gate-in is inside the appointment window. A gate-in after `window_start_utc + carrier_cutoff_min` is stale even if it is still inside the appointment window, but a gate-in exactly at `window_start_utc + carrier_cutoff_min` is not past the cutoff. If the latest gate-in is outside the appointment window, use `OUTSIDE_WINDOW` and do not also add `CARRIER_CUTOFF_MISSED`. A container's display event is the latest event by timestamp. Its gate-in status is based on the latest `gate_in` event by timestamp. A hold is active only when the latest `hold` event is newer than the latest `release` event. When events for the same container have the same timestamp, treat the later line in `events.ndjson` as the later event for display-event, gate-in, and hold/release decisions. Missing appointments are `unknown_appointment`, active holds are `blocked`, missing or out-of-window gate-ins are `stale`, in-window gate-ins after the carrier cutoff are `stale`, refrigerated containers whose latest `temp_check` at or before gate-in is missing or more than six hours old are `blocked`, bookings with more non-blocked in-window containers than capacity mark all of those non-blocked in-window containers as `overbooked`, and the rest are `ready`. A `temp_check` exactly six hours before gate-in is still fresh; a `temp_check` after gate-in cannot satisfy the freshness requirement. Keep the reason codes stable: `NO_APPOINTMENT`, `ACTIVE_HOLD`, `NO_GATE_IN`, `OUTSIDE_WINDOW`, `CARRIER_CUTOFF_MISSED`, `TEMP_CHECK_MISSING`, and `CAPACITY_EXCEEDED`.

The `bookings` audit summary must include one entry for each booking that appears in the manifest, sorted by `booking`. Each entry must contain `booking`, `capacity`, `appointment_window_utc`, `container_ids`, `carriers`, and `status_counts`. `container_ids` must be sorted, and `carriers` must be sorted and deduplicated so each carrier appears at most once per booking. `status_counts` must use the same five status keys as the top-level summary and count only containers for that booking. For bookings with no appointment, use `capacity: 0` and `appointment_window_utc: []`.

The `carrier_summary` audit summary must include one entry for each carrier that appears in the manifest, sorted by `carrier`. Each entry must contain `carrier`, `booking_ids`, `container_ids`, `requires_temp_check`, and `status_counts`. `booking_ids` and `container_ids` must be sorted and deduplicated as appropriate. `requires_temp_check` is the count of manifest containers for that carrier whose `requires_temp_check` value is true. `status_counts` must use the same five status keys as the top-level summary and count only containers for that carrier.

The `exceptions` ledger must include every non-`ready` container and no `ready` containers. Sort exceptions first by status severity in this exact order: `blocked`, `overbooked`, `stale`, `unknown_appointment`; break ties by `container_id`. Each exception entry must contain `container_id`, `booking`, `carrier`, `status`, `primary_reason`, `last_event_utc`, and `minutes_from_window_start`. `primary_reason` is the first reason code for that container, or an empty string if there is no reason code. `minutes_from_window_start` is the integer number of elapsed minutes from the appointment start to the latest gate-in event; use `null` when the container has no appointment or no gate-in event.
