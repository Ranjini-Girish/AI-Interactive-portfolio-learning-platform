The TypeScript on Node 22 FSM transition auditor at `/app` compiles but writes wrong metrics and misses findings. Fix the crate (do not rewrite in another language), build with `cargo build --release`, install the binary at `/app/build/fsm-transition-audit`, and run it to produce `/app/output/fsm_audit_report.json`. Do not modify `/app/data/` or `/app/config/`.

Workflow logs live in `/app/data/workflows/`; FSM definitions in `/app/data/fsm_defs/`; severities in `/app/config/policy.json`. `/app/docs/` gives domain context; this instruction is authoritative.

Within each workflow, sort transitions by `(sequence ASC, logged_at ASC)`. Walk in order: keep the first occurrence of each `transition_id`; later duplicates are skipped with finding `duplicate_transition_skipped` (evidence `duplicate_sequence`). Track `current` state starting at the FSM `initial_state`. For each kept transition: if `from_state != current`, emit `state_mismatch` (evidence `expected_state`, `from_state`); else if `current` is a terminal state, emit `terminal_reopened`; else if `(current, to_state)` is not in `allowed_edges`, emit `illegal_transition`; then append the transition and set `current = to_state`. After the walk, if any kept transition has `logged_at` strictly less than the previous kept transition, emit `timestamp_regression`. If final `current` is not in `terminal_states`, emit `stuck_workflow` (evidence `final_state`).

`avg_dwell_ms` per workflow is the harmonic mean of `duration_ms` over kept transitions with `duration_ms > 0` (round 6 decimals; 0.0 if none). `states_visited` counts distinct states reached including `initial_state`. Summary `avg_workflow_dwell_ms` is the harmonic mean of per-workflow `avg_dwell_ms` values that are `> 0`.

`source_hashes` maps `data/workflows/*.json` and `data/fsm_defs/*.json` to SHA-256 hex of file bytes after CRLF→LF normalization and stripping one trailing newline if present. `integrity_hash` is SHA-256 hex of UTF-8 lines `workflow_id|transition_id|sequence|to_state` joined with `\n` (no trailing newline before hash): workflows in ascending `workflow_id`, transitions in replay order within each workflow (not sorted by `transition_id`).

Report JSON: `schema_version` 1; top-level keys `findings`, `source_hashes`, `summary`, `workflow_audits` (alphabetically sorted keys at every object). `workflow_audits` sorted by `workflow_id`; each entry has `workflow_id`, `fsm_id`, `transitions_kept`, `transitions_skipped`, `final_state`, `states_visited`, `avg_dwell_ms`, `integrity_lines`. `findings` sorted by `(severity_rank, finding_type, workflow_id, transition_id)` with null `transition_id` before non-null. `summary` includes `workflow_count`, totals, `findings_by_type`, `findings_by_severity` (all five severity keys, zero if none), `avg_workflow_dwell_ms`, `integrity_hash`. Use 2-space indent and trailing newline.
