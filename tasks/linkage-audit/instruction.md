You are given a Java 21 project under `/app/project/` containing headers in `include/`, source files in `src/`, and four JSON configuration files at the project root. Implement the analysis described below as a Java 21 program, compile it to `/app/build/analyzer`, and run it to produce `/app/output/audit_report.json`. The environment has `javac` and `make` installed; acquire any additional libraries you need. Do not modify any file under `/app/project/`.

Parse `#include "..."` directives from every `.h` and `.cpp` file under `/app/project/`. Ignore `#include <...>` system includes. Resolve quoted include paths relative to `/app/project/include/`. For each compilation unit (`.cpp` file), compute its transitive include closure: recursively follow `#include "..."` directives in each included header, deduplicating any header that contains an include guard (the `#ifndef IDENTIFIER` / `#define IDENTIFIER` / `#endif` pattern at file boundaries). A guarded header appears at most once in the closure regardless of how many inclusion paths reach it. Report `direct_includes` (the headers the `.cpp` directly includes, sorted ascending), `transitive_closure` (all unique reachable headers sorted ascending), `include_depth` (0 if no includes; otherwise 1 plus the maximum include_depth among directly-included headers, where a header's depth follows the same recursive rule), and `n_effective_headers` (the length of `transitive_closure`).

Read `/app/project/symbol_registry.json`. Each key is a source path relative to `/app/project/`; each value has `defines` (array of `{"symbol", "linkage"}` with linkage `strong`, `weak`, or `local`) and `references` (array of symbol name strings). Local symbols are invisible outside their compilation unit and never enter the global defined set. Weak symbols (e.g. inline functions defined in headers) may coexist across multiple compilation units without conflict.

For each target in `/app/project/build_targets.json`, simulate link resolution with the single-pass archive model. Initialize `defined` and `unresolved` as empty sets. Process every file in `objects` (in listed order): add its strong and weak defined symbols to `defined`, then for each header in its transitive include closure add that header's weak symbols (from the registry) to `defined`, then add every reference not already in `defined` to `unresolved`. Next, process `libraries` in listed order. For each library, iterate its `members` in listed order: if the member defines (strong or weak) any symbol currently in `unresolved`, include it — add its strong and weak defines to `defined`, remove newly defined symbols from `unresolved`, add weak header symbols from its transitive closure to `defined` and remove any from `unresolved`, and add its references not in `defined` to `unresolved`; otherwise skip the member. After all libraries, symbols remaining in `unresolved` are unresolved externals. Report `included_units` (all objects plus pulled-in members, sorted), `skipped_members` (library members not pulled in, sorted), `unresolved_symbols` (sorted), `odr_violations` (list of `{"symbol", "units"}` for each symbol defined as `strong` by two or more included compilation units; `units` sorted ascending), and `status` (`resolved` if unresolved is empty, otherwise `unresolved`).

Read `/app/project/dirty_files.json` (a list of paths relative to `/app/project/`). A compilation unit needs recompiling if it appears in `dirty_files` or any header in its transitive closure appears there. A target needs relinking if any of its included units (from the link resolution above — not all library members, only the ones actually pulled in or listed as objects) needs recompiling. Report `dirty_files` (as given), `recompile_set` (sorted compilation unit paths), and `relink_targets` (sorted target names).

Emit `quality_findings`. For each target: one `odr_violation` finding per duplicated strong symbol; one `unresolved_symbol` finding per unresolved symbol; one `unused_library_member` finding per skipped library member. Severities come from `/app/project/policy.json` field `finding_types`; ranks from `severity_ranks`. Each finding has `finding_type`, `severity`, `severity_rank`, `target`, `library` (library name for unused members, else null), `symbol` (for odr/unresolved, else null), `member` (source path for unused members, else null), and `units` (sorted source paths for odr violations, else null). Sort findings by `(severity_rank ASC, finding_type ASC, target ASC, coalesce(symbol, member, "") ASC)`.

All paths in the report are relative to `/app/project/`. Output a single JSON object at `/app/output/audit_report.json` with two-space indent and trailing newline. Top-level keys in this order: `schema_version` (integer 1), `summary`, `include_analysis` (sorted by `compilation_unit`), `link_resolution` (sorted by `target`), `rebuild_impact`, `quality_findings`. The `summary` object has `total_compilation_units`, `total_headers`, `total_targets`, `targets_resolved` (number of targets whose link resolution `status` is `resolved`), `targets_with_issues` (number of targets whose link resolution `status` is not `resolved`, i.e. `total_targets - targets_resolved`), `total_findings`, `by_severity` (all five keys `critical`/`high`/`medium`/`low`/`info` always present, zeros for absent), `by_finding_type` (only types with count > 0), `recompile_count`, `relink_count`.
