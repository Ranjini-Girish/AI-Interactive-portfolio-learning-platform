The TypeScript on Node 22 scope-guard auditor at `/app` builds but reports wrong metrics and misses findings. Fix the crate in place (no TypeScript on Node 22 rewrite), run `cargo build --release`, install the binary at `/app/build/scope-guard-audit`, and emit `/app/output/scope_audit_report.json`. Leave `/app/data/` and `/app/config/` unchanged.

Scope event logs are under `/app/data/workloads/`; severities in `/app/config/policy.json`. `/app/docs/` is background only—this instruction defines required behavior.

Per workload, order events by `(sequence ASC, logged_at ASC)`. Keep the first row per `event_id`; later duplicates are skipped with finding `duplicate_event_skipped` (evidence `duplicate_sequence`). Maintain a LIFO stack of `scope_id` values: `enter` pushes; `exit` must pop the top and matches `scope_id` or emit `scope_exit_mismatch` (evidence `expected_scope`, `scope_id`); empty-stack `exit` is `exit_without_enter`. `enter` while `scope_id` is already on the stack emits `scope_reentered`. `panic_mark` pins the current top scope—until that scope is exited, an `exit` for a different active scope also emits `unwind_order_violation` (evidence `panic_scope`, `scope_id`). After replay, each scope still on the stack yields `scope_leak` (evidence `scope_id`). If a kept event’s `logged_at` is strictly before the previous kept event’s, emit `timestamp_regression`.

`avg_hold_ms` per workload is the harmonic mean of `hold_ms` on kept `exit` events with `hold_ms > 0` (six decimal places; 0.0 if none). `max_stack_depth` is the maximum stack size during replay. Summary `avg_workload_hold_ms` is the harmonic mean of per-workload `avg_hold_ms` values that are `> 0`.

`source_hashes` maps each `data/workloads/*.json` path to SHA-256 hex of file bytes after CRLF→LF normalization and removing one trailing newline when present. `integrity_hash` is SHA-256 hex of UTF-8 lines `workload_id|event_id|sequence|kind|scope_id` joined with `\n` (no trailing newline before hashing): workloads sorted by `workload_id`, events in replay keep-order.

Report JSON uses `schema_version` 1; top-level keys `findings`, `source_hashes`, `summary`, `workload_audits` (alphabetically sorted object keys everywhere). `workload_audits` sorted by `workload_id`; each entry includes `workload_id`, `events_kept`, `events_skipped`, `max_stack_depth`, `scopes_leaked`, `avg_hold_ms`, `integrity_lines`. `findings` sorted by `(severity_rank, finding_type, workload_id, event_id)` with null `event_id` before non-null. `summary` includes `workload_count`, event totals, `findings_by_type`, `findings_by_severity` (all five severity keys, zero if none), `avg_workload_hold_ms`, `integrity_hash`. Two-space indent and trailing newline.
