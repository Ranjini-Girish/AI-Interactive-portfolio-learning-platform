`lock_order_graph.json` carries `{cycles, edges, nodes, topological_layers}` over the **lock-order graph** induced by the trace history: a node per lock id (sorted ASCII), and an edge `A → B` for every ordered pair such that some thread held `A` (in any mode) at the moment it issued an `acquire` for `B` that succeeded **or** blocked (deduplicated across the whole trace). `edges` are `{first_witness_seq, from, to}` triples sorted by `(from, to)`, where `first_witness_seq` is the smallest `seq` of any event that established the edge. `nodes` carry `{id, in_degree, out_degree}` from the deduplicated edge set, sorted by id. `cycles` are the multi-vertex SCCs of this graph (each sorted ASCII inside, outer list sorted by lex-smallest member); these are the **lock-order inversions** that risk future deadlocks. `topological_layers` are the Kahn-style layers of the SCC condensation flattened to ASCII-sorted lock ids per layer in layer-index order. `thread_diagnostics.json` carries `threads` (sorted by id) of `{diagnostics, id}` where `diagnostics` is the sorted-by-`(severity_rank, code, seq)` list of `{code, lock, seq, severity}` entries with `severity` ∈ {"error","warning","note"} (ranks `error<warning<note`), `lock` either a lock id or null, and `code` drawn from the fixed set `{"E_UNMATCHED_RELEASE","E_NON_REENTRANT_REACQUIRE","E_UNSUPPORTED_MODE","E_WAIT_WHILE_HOLDING","E_DEADLOCKED_AT_END","W_HELD_AT_EXIT","W_LOCK_ORDER_INVERSION","N_BLOCKED_ON_ACQUIRE"}`. The `lock` field is a lock id for every code EXCEPT `E_DEADLOCKED_AT_END`, which is the sole **thread-only** code and MUST always carry `lock = null` (the diagnostic is about the thread being stuck at trace end, not about any single lock — even when the thread is waiting on exactly one lock, you must still emit `lock: null`). `E_DEADLOCKED_AT_END` fires for every thread that participates in any `wait_for_graph` cycle and its `seq` field MUST equal the maximum `seq` across every wait-queue entry whose `thread` is this thread at trace end (i.e. `max(w["seq"] for lk in locks for w in lk.wait_queue if w["thread"] == t)`); if for any reason the thread has no wait-queue entry (which cannot normally happen for a thread that participates in a `wait_for_graph` cycle), use `0`. Do NOT use the last-event seq, the seq of the cycle-witnessing edge, `INT64_MAX`, or any other sentinel — only the max wait-queue seq is correct. `W_HELD_AT_EXIT` fires for every `(thread, lock)` pair where the thread issued an `exit` event while still holding the lock (one diagnostic per such lock); `W_LOCK_ORDER_INVERSION` fires once per `(thread, lock)` pair when the thread first issued an `acquire` for `lock` while holding some other lock that participates in any `lock_order_graph` cycle with `lock`; `N_BLOCKED_ON_ACQUIRE` fires once per `(thread, lock, seq)` whenever the thread's acquire blocked (whether it was eventually granted or not). When `enforce_strict_order` is false, `W_LOCK_ORDER_INVERSION` is suppressed entirely and the diagnostic does not appear. `contention_summary.json` carries `{hot_locks, longest_holds, summary}` where `summary` has `{deadlocked_threads, threads_total, locks_total, total_acquires, total_blocked_acquires, total_releases, total_unmatched_releases}` (every count is an integer). `summary.total_acquires` counts EVERY `acquire` event in the trace (granted, blocked-and-queued, recursive re-entries, AND `E_UNSUPPORTED_MODE`/`E_NON_REENTRANT_REACQUIRE`/duplicate-queued early-exits) — increment it at the top of acquire-event handling before any early-exit check; `summary.total_blocked_acquires` counts only those acquires that were actually appended to a wait queue; `summary.total_releases` counts only `release` events that found the thread holding the lock and decremented (or fully released) its hold (silent unmatched releases under `unmatched_release_action == "ignore"` and unmatched releases under `"error"` do NOT count as releases — they count as `total_unmatched_releases` instead). Per-lock `hot_locks[*].acquire_count` follows the strictly more restrictive rule below (it excludes ALL FOUR early-exit branches: `E_UNSUPPORTED_MODE`, `E_NON_REENTRANT_REACQUIRE`, recursive re-entry counter increments, AND duplicate-queued silent skips), so `summary.total_acquires` is always greater than or equal to `sum(hot_locks[*].acquire_count)`, and is strictly greater whenever the trace exercises any of those four early-exit branches. `hot_locks` is the sorted-by-`(-acquire_count, id)` list (so the most-acquired locks come first, with ASCII id ASCENDING as tiebreaker) of `{acquire_count, blocked_count, id, peak_holders}` for every lock whose `acquire_count` is at least 1, where `acquire_count` counts ONLY acquire events that successfully reached the holder/wait-queue update step — concretely: (a) acquires that were granted immediately (added to the holder set), and (b) acquires that blocked and were appended to the wait queue — and NOTHING ELSE. The four branches that DO NOT contribute to `acquire_count` are: (1) `E_UNSUPPORTED_MODE` rejections, (2) `E_NON_REENTRANT_REACQUIRE` rejections, (3) recursive re-entries on a recursive lock that only incremented the per-thread hold counter (these advance `recursive_holds[thr]` and short-circuit BEFORE the holder/queue update step, so they DO NOT contribute to `acquire_count`), and (4) silently-ignored duplicates of an acquire already sitting in the wait queue. Equivalent reference rule (matches the verifier byte-for-byte): increment `acquire_count` only inside the same code path that either (i) calls the "add holder" routine, or (ii) appends a new entry to `wait_queue`; never increment it on the four early-exit branches above. Concrete worked example: if a thread issues `acquire(L)` at seq 0 (granted), `acquire(L)` again at seq 1 on a `recursive=true` lock (counter goes 1→2 — DO NOT count for `acquire_count`), then later `acquire(L)` at seq 5 (blocked and queued — count it), `L`'s `acquire_count` is `2`, not `3`; `summary.total_acquires` for the same trace gets `+3` from these three events (it counts all acquire events including the recursive re-entry). `blocked_count` counts how many of those counted acquires blocked at issue time (i.e. were appended to the wait queue), and `peak_holders` is the maximum number of distinct concurrent holders the lock ever had. `longest_holds` is the top-N list (`N = min(5, total_distinct_hold_intervals)`) of the longest **inclusive** `(release_tick - acquire_tick)` durations across all completed hold intervals (a thread holding a lock multiple times produces one interval per recursion-zero release), sorted by `(-duration, lock, thread, acquire_seq)` and emitted as `{acquire_seq, duration, lock, release_seq, thread}` — `duration` is computed in ticks and is a non-negative integer; release events that never closed (still held at end) are excluded from `longest_holds`. `safe_order_plan.json` carries `{cycles, layers, plan}`: `plan` is a **single flat ASCII list of strings** (`list[str]`, NOT a list-of-lists) — it is the Kahn-style topological order of the lock-order graph's SCC condensation, produced by concatenating the layers in layer-index order with each layer's locks sorted ASCII-ascending before append (so within a tied Kahn layer the locks appear in ASCII order, and across layers the earlier layers come first); `layers` is the same data shaped as a `list[list[str]]` (one inner list per Kahn layer, ASCII-sorted inside, in layer-index order); and `cycles` repeats `lock_order_graph.cycles` as the canonical list of locks that cannot be safely ordered against each other.
