A multi-tenant TLS certificate inventory is spread across the JSON files under `/app/certs/`: `pool_state.json`, `chain_config.json`, `ocsp_responses.json`, `incident_log.json`, per-tier domain specs at `domains/<tier>/<domain>.json`, and synthetic certificate metadata at `leafs/<serial>.json` and `intermediates/<serial>.json`. The complete contract — input validation rules, chain-walk semantics, OCSP staleness rules, tier-dependent SAN matching, validity windows, the compromise cascade, and the exact schema of every required artifact — is documented in `/app/certs/SPEC.md` (also linked at `/app/docs/tls_cert_chain_spec.md`); read that file before producing output. Write exactly five JSON files into `/app/audit/`: `chain_audit.json`, `expiry_report.json`, `ocsp_summary.json`, `ca_risk.json`, and `summary.json`. The dataset under `/app/certs/` is read-only and must not be modified.

All outputs must be UTF-8, indented with two spaces, with object keys emitted in sorted order at every level, and end in a single trailing newline; every list field must follow the sort order specified in `/app/certs/SPEC.md`. Record leaf OCSP per domain in `ocsp_summary.json`, but roll up worst-of-chain OCSP into `summary.by_ocsp_state` as defined in the spec; those two tallies are not interchangeable.
