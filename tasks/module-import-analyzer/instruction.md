A TypeScript ES module codebase lives at `/app/data/src/` containing 21 source files organized into a layered architecture with handlers, services, and utilities. Entry points are specified in `/app/data/entry-points.json`. Write a TypeScript on Node 22 script that performs static import/export analysis on all `.js` files and writes the result to `/app/output/module_analysis.json`.

The analysis must parse every import and export statement using pattern matching (no external parser dependencies). Import types to detect: named (`import { x } from`), default (`import x from`), namespace (`import * as x from`), and side-effect (`import './x'`). Export types: named declarations, default exports, re-export named (`export { x } from`), re-export default-as-named (`export { default as Y } from`), and re-export all (`export * from`). Only internal modules with relative paths are analyzed; bare specifiers are ignored.

The output JSON must contain: a `dependency_graph` mapping each module to its sorted dependency list (including re-export sources); a `modules` object with each module's sorted export symbols, import details, import count, and reachability from entry points; `cycles` detected as strongly connected components of size two or more (each cycle sorted, outer array sorted); `unused_exports` listing symbols exported but never imported by any module (namespace imports use all named exports, re-exports count as uses of source symbols, `export *` re-exports only named exports not default); `unreachable_modules` not transitively reachable from entry points; and a `summary` with counts and `max_depth` as the longest BFS shortest-path from entry points.

Output format: 2-space indentation, keys sorted alphabetically at all levels, trailing newline. Reference `/app/docs/OUTPUT_FORMAT.md` for the complete schema and `/app/docs/IMPORT_RULES.md` for pattern details.
