# Spectral Calibration Audit

We've got a stack of spectroscopy calibration runs under `/app/experiments/` and need a calibration audit out the other side. Read everything beneath that directory as inputs and write `run_audit.json`, `calibration_summary.json`, and `peak_table.csv` to `/app/output/`; treat `/app/experiments/` as strictly read-only. A C++17 starter at `/app/environment/calibrate.cpp` builds via `make -C /app/environment build`; for the actual implementation put your sources under `/app/src/`, ship a top-level `/app/Makefile`, and have `make` from `/app` produce the binary at `/app/build/calibrate`. The binary takes no arguments, runs from any working directory, and resolves both inputs (`$APP_ROOT/experiments/`) and outputs (`$APP_ROOT/output/`) from the `APP_ROOT` environment variable (defaulting to `/app`). Re-running on the same inputs must produce byte-identical output files; the harness rejects any `*.py` file under `/app` outside `experiments/`.

`manifest.json` lists each run and the spectrum CSV that backs it; `config/instruments.json`, `config/batches.json`, and `config/calibrants.json` carry shared metadata and the expected peak windows; spectra are CSVs with header `wavelength_nm,intensity` where rows whose either field isn't finite are skipped. A run's effective metadata stacks `instruments.json`, then `batches.json`, then the manifest entry, with later layers overriding `operator`, `temperature_c`, `quality_flag`, `calibrant`, and `wavelength_offset_nm`. Drop a run with reason `missing_metadata` when any required inherited field is absent or its spectrum file is missing, `quality_flag` when the inherited flag isn't `"ok"`, `insufficient_rows` when fewer than 8 valid spectrum rows survive, `no_baseline` when nothing survives the baseline filter, or `missing_peak` when any expected peak has no positive baseline-corrected sample inside its window. For each surviving run the off-peak pool is every spectrum row whose wavelength sits outside every expected peak window; the baseline is the arithmetic mean of off-peak rows that survive a 3·MAD filter around the off-peak median (when the MAD is zero, keep only rows equal to the median). Report the Rousseeuw–Croux Qn scale estimator over that pool — asymptotic form, constant 2.21914 — as `noise_floor_qn`, and the sample variance with n−1 denominator over every valid intensity in the run as `intensity_var`. Each peak's centroid is the intensity-weighted mean wavelength over rows with positive baseline-corrected intensity inside its window; subtract `wavelength_offset_nm` to get `observed_nm_raw`, set `peak_area` to the sum of those positive baseline-corrected intensities, and `raw_error_nm = observed_nm_raw − expected_nm`. Once every successful run's raw peaks exist, fit a per-`(calibrant, instrument)` drift trend `raw_error_nm ≈ slope · expected_nm + intercept` by intensity-weighted least squares with `peak_area` as each point's weight, then iteratively reject any peak whose absolute residual exceeds 2.5 times the weighted population residual standard deviation and refit, capped at five weighted fits where the initial fit before any rejection counts as iteration 1; if the kept set ever has fewer than two distinct `expected_nm` values the slope is zero and the intercept is the weight-weighted mean of the kept y's (zero when the kept set is empty). Apply the converged slope and intercept to every peak in the group — including ones the loop rejected — to produce drift-corrected `observed_nm = observed_nm_raw − (slope · expected_nm + intercept)` and `error_nm = observed_nm − expected_nm`, and track `iterations_used` and `n_outliers_removed`.

`run_audit.json` has three top-level keys: `included_runs` sorted by `run_id`, holding objects with `run_id`, `instrument`, `batch`, `operator`, `calibrant`, `temperature_c`, `wavelength_offset_nm`, `baseline`, `noise_floor_qn`, `intensity_var`, `rms_error_nm` (root mean square of the run's unrounded drift-corrected per-peak errors), and `status: "included"`; `excluded_runs` sorted by `run_id`, holding `run_id`, `status: "excluded"`, and one of the five `reason` strings above; and `source_files`, the sorted ASCII-ascending list of every regular file beneath `/app/experiments/` expressed with the literal `/app/experiments/…` prefix. `calibration_summary.json` has `calibrants` sorted by `(calibrant, instrument)`, holding `calibrant`, `instrument`, `run_count`, `peak_count`, `mean_error_nm`, `rms_error_nm` (both aggregated across every drift-corrected peak in the group, including ones the loop rejected), `residual_stddev_nm` (weighted residual stddev of the final fit over the kept peaks only), `n_outliers_removed`, and `iterations_used`; and `best_run_by_calibrant` sorted by `calibrant`, holding `calibrant`, `run_id`, `instrument`, and `rms_error_nm` — one per calibrant, picked as the run that ranks first by ascending `(rms_error_nm rounded to 4 decimals, max absolute drift-corrected error, run_id)` with the chosen run's full 6-decimal `rms_error_nm` carried through. `peak_table.csv` uses header `run_id,instrument,batch,calibrant,peak_label,expected_nm,observed_nm,error_nm,peak_area`, rows sorted by `(run_id, peak_label)`, with drift-corrected `observed_nm` and `error_nm`. Both JSON outputs use 2-space indentation; numeric output fields round to 6 decimals (the 4-decimal rounding lives only inside the tie-break key); CSV numbers carry up to 6-decimal precision and round-trip equal to the JSON values when re-parsed as floats.
