Fix the bugs in the sensor data aggregation tool at `/app/`. Build with `make -C /app` and run `/app/build/sensor_tool`. It reads CSV sensor files from `/app/data/`, applies pipeline config from `/app/config/pipeline.json`, and writes `/app/output/report.json`.

Each CSV has columns: timestamp (ISO 8601 UTC), sensor_id, reading (float), unit, quality. The tool filters records whose quality is in config.quality_filter, groups each sensor's filtered readings into time buckets of config.bucket_size_seconds seconds where bucket_start = floor(epoch_seconds / bucket_size) * bucket_size using half-open intervals [start, start+size), computes per-bucket mean, min, max, sample standard deviation (Bessel-corrected, divisor N-1), and count, then detects threshold violations by matching sensor IDs against rule patterns (prefix before the *) and checking if the named metric exceeds the threshold (gt means >, lt means <).

Sensor summaries are sorted by config.sort_by in config.sort_order (ascending = lexicographic a<b). All floats use config.output_precision decimal places. The integrity hash is "sha256:" followed by the hex SHA-256 of the JSON-serialized sensor_summaries array only — not metadata or integrity.

Output JSON keys: metadata (total_records, filtered_records, sensors_processed, time_range with start/end ISO timestamps, config_hash as SHA-256 hex of the raw config file), sensor_summaries (array with sensor_id, record_count, buckets array with start/end/mean/min/max/stddev/count, violations array with bucket_start/metric/value/threshold), integrity (results_hash). Use 2-space indentation with trailing newline. Do not modify `/app/data/` or `/app/config/`. Solution must be compiled Java 21.
